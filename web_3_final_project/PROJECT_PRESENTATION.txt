SLIDE 1: PROJECT OVERVIEW
================================================================================

Azure Luxe Hotel
A hotel booking and management system

What it does:
• Public room browsing and booking
• Guest booking lookup without login
• Customer self-service dashboards
• Staff operational management
• Admin oversight and controls

Built with Laravel 12 + Blade templating

SLIDE 11: SERVICE LAYER
================================================================================

Reusable Business Logic & Transactions

BookingService
  • checkAvailability(roomId, checkIn, checkOut)
    → Validates room status + date overlap checks
  
  • calculatePrice(roomId, checkIn, checkOut, promoCode)
    → Days × base price - discount
    → Returns calculated total
  
  • validatePromoCode(code)
    → Checks active status and expiry
    → Returns promo code object or null
  
  • createBooking(data)
    → Database transaction wrapper
    → Validates availability
    → Calculates price
    → Creates booking record
    → Updates room status if walk-in

RoomService
  • getAvailableRooms(checkIn, checkOut, categoryId)
    → Returns rooms available for date range
    → Filters by category if specified
  
  • updateRoomStatus(roomId, status)
    → Sets room to available, occupied, or dirty

PromoCodeService
  • validate(code)
    → Returns valid promo code or null
  
  • applyDiscount(originalPrice, promoCode)
    → Calculates and returns discounted price

SLIDE 13: BOOKING TRANSACTION FLOW
================================================================================

From Booking Form to Confirmation

User (Guest or Customer) Fills Booking Form
  ↓
StoreBookingRequest Validates Input
  ↓
BookingController Receives Valid Data
  ↓
BookingService::createBooking() Begins Transaction
  
  1. checkAvailability()
     • Room status = available?
     • Date overlap with pending/confirmed/checked_in bookings?
     
  2. calculatePrice()
     • Days = check_out - check_in
     • Total = base_price × days
     • Apply promo discount if code valid
     
  3. Create Booking Record
     • Save all booking data to database
     • user_id = null for guests, user ID for customers
     
  4. Update Room State
     • If walk-in: set room to occupied
     • Otherwise: room stays available (guest not checked in yet)
     
  5. Commit Transaction / Clear Cache
  
  ↓
Response Depends on Caller Type
  • Guest → Redirect to booking confirmation page
  • Customer → Redirect to /customer/bookings
  • AJAX → JSON response with booking ID

SLIDE 14: ROOM STATUS LIFECYCLE
================================================================================

Room States & Transitions

AVAILABLE (initial state)
  ├─ Pending booking created? → stays AVAILABLE
  └─ Walk-in booking checked-in? → becomes OCCUPIED

OCCUPIED
  └─ Guest checked-out? → becomes DIRTY

DIRTY
  └─ Staff marks room cleaned? → becomes AVAILABLE

Special Cases:
• Booking cancelled/no-show with no other active bookings → room AVAILABLE
• Multiple overlapping bookings → room OCCUPIED until all checked-out
• Pending/confirmed bookings don't change room status automatically

SLIDE 16: KEY RELATIONSHIPS
================================================================================

Entity Relationships & Data Flow

User → has many → Booking
  • A customer can have multiple bookings
  • Guests have bookings with user_id = null

RoomCategory → has many → Room
  • Category defines base price and capacity
  • Multiple rooms per category (e.g., 10 "Deluxe" rooms)

Room → has many → Booking
  • A room can have multiple bookings (different dates)
  • Status reflects current or most recent occupancy

Booking → belongs to → User (nullable)
  • Links to customer account if authenticated
  • NULL for guest bookings

Booking → belongs to → Room
  • Each booking tied to exactly one room

PromoCode (standalone)
  • Referenced by booking.promo_code string (not FK)
  • Used during price calculation
